# タイトル

Reading @- / @+ after a utf8 match is orders of magnitude slower than pos()
（utf8 文字列のマッチ後に @- / @+ を読むのは pos() より桁違いに遅い）

# 本文

## Description（説明）

多バイト文字を含む文字列でマッチに成功した後、`$-[n]` / `$+[n]` を
読むと、マッチのバイトオフセットに比例した時間がかかります。読むた
びに、正規表現エンジンが保存した文字列（`subbeg`）の先頭から
`utf8_length()` で文字数を数え直すためです。何もキャッシュされない
ので、各マッチでオフセットを読み取るループ — マッチ位置を集める普
通の書き方 — が文字列全体に対して二次関数的になります。

同じ情報を `pos()` 経由で得ると 2 桁安く済みます。pos() は SV の
UTF-8 位置キャッシュ（`sv_pos_b2u`）を通して変換し、そちらは正常に
機能しているからです。

これは退行ではなく、昔からこうです（最初に測ったのは 5.12〜5.16 の
時代でした）。#24531 と関連しますが、あちらは逆方向の変換の話で、
別の問題です。

## Steps to Reproduce（再現手順）

```perl
use Time::HiRes qw(time);
my $s = (chr(0x3042) . chr(0x3044) . chr(0x3046) . " abc ") x 12_000;
my($t, $c);

$t = time; $c = 0;
while ($s =~ /abc/g) { my($b, $e) = ($-[0], $+[0]); $c++ }
printf "match vars: %7.3f sec (%d matches)\n", time - $t, $c;

$t = time; $c = 0;
while ($s =~ /abc/gp) { my($b, $e) = (pos($s) - length(${^MATCH}), pos($s)); $c++ }
printf "pos():      %7.3f sec (%d matches)\n", time - $t, $c;
```

5.42.2（ubuntu-latest）での結果：

```
match vars:   0.560 sec (12000 matches)
pos():        0.003 sec (12000 matches)
```

5.12.5 から 5.42.2 までの全リリースをベンチマークしました
（[結果とワークフロー](https://github.com/kaz-utashiro/perl-matchvars-bench)）：

| perl | `@-`/`@+`（秒） | pos()（秒） | 比率 |
|---|---:|---:|---:|
| 5.12.5 | 1.45 | 0.153 | 9 倍 |
| 5.14.4 – 5.16.3 | 2.9 – 3.3 | 0.15 | 20 倍 |
| 5.18.4 | 6.5 | 0.068 | 95 倍 |
| 5.20.3 – 5.36.3 | 5.7 – 8.4 | 0.003 | 約 2000 倍 |
| 5.38.0 – 5.42.2 | 0.46 – 0.57 | 0.003 | 150 – 190 倍 |
| blead（ソースからビルド） | 0.59 | 0.003 | 222 倍 |

数字から歴史が読み取れます。`pos()` 経路も元は遅かったのですが、
5.18/5.20 で修正され、以来ずっと高速です。マッチ変数の方は一度も直
されていません。5.14 と 5.18 で遅くなり、5.38 で約 12 倍改善しまし
たが（おそらく `utf8_length()` 自体の高速化によるもので、定数が変
わっただけで計算量は不変）、今日でも読み取り 1 回あたり O(オフセッ
ト) のままです。

## Analysis（分析）

mg.c の `Perl_magic_regdatum_get()`：

```c
if (RX_MATCH_UTF8(rx)) {
    const char * const b = RX_SUBBEG(rx);
    if (b)
        i = RX_SUBCOFFSET(rx) +
                utf8_length((U8*)b,
                    (U8*)(b-RX_SUBOFFSET(rx)+i));
}
```

読むたびに、バイトオフセット→文字オフセットの変換を `subbeg` の先
頭から数えることで毎回行っています。この変換は元の SV ではなく正規
表現側の保存コピーに対するものなので、SV の UTF-8 位置キャッシュは
使えません。pos()（SV のキャッシュを通る）が速く、この経路が遅いの
は、まさにこの構造的な違いによります。

考えられる修正は、regexp 構造体に小さな（バイトオフセット, 文字オ
フセット）キャッシュを持たせることです。新しいマッチが `subbeg` を
埋めたときにリセットし、キャッシュ位置から差分だけ歩く。典型的なア
クセスパターン（$-[0] → $+[0] → 次のマッチのオフセット、と昇順に
進む）なら、マッチループ全体で償却 O(文字列長) となり、pos() イディ
オムと同等になります。構造体へのフィールド追加になるので、次の開発
サイクル向けの変更です。

## Real-world impact（実世界での影響）

#24531 と同じ背景です。[App::Greple](https://metacpan.org/dist/App-Greple)
はこの問題のためにマッチ変数を一切使わず、`pos()` + `${^MATCH}` を
10 年以上使い続けています。大きな多バイト文字列に対して `@-`/`@+`
で素朴にマッチ位置を集めるコードは、理由の分からないまま二次関数的
なコストを払うことになります。
